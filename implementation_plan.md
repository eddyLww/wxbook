# 深度分析与解决方案：解决“精读内容尚未就绪”问题

经过深度代码追溯与架构分析，发现导致点击“精读摘要”频繁提示“内容尚未就绪”的核心问题，不仅是单纯的“数据缺失”，更深层是由**云函数超时机制**与**AI大模型生成耗时**之间的矛盾导致的。

## 🔍 深度病理分析 (Root Cause Analysis)

触发该 Bug 的完整调用链坍塌过程如下：

1. **AI 生成耗时极长 vs 云环境极短的生命周期**：
   - 客户端调用 `bookService` -> 触发 `aiGenerator` -> 请求 DeepSeek API。
   - DeepSeek 生成上千字的精读内容往往需要 **15 - 40 秒**。
   - 然而，微信小程序的**默认云函数超时时间只有 3 秒**！此外，哪怕增加了云端超时，**客户端（前端）默认也有最大 15-20 秒的请求等待时长限制**。
2. **请求超时导致的状态异常**：
   - 当请求时间超过阈值，前端 `detail.js` 会触发 `catch(err)`。
   - 前端把加载状态（`aiLoading`）强行置为 `false`，但因为后端异常，拿到的是一个**没有 `aiFullContent` 字段的“半成品书籍对象”**。
3. **“死亡循环”与“多重并发”灾难**：
   - 用户看到 `aiLoading` 圈圈消失了，以为加载好了，点击“精读摘要”。
   - 代码检查发现 `!this.data.book.aiFullContent` 为真，弹出“尚未就绪”的提示，并且**再次触发 `fetchBookDetail` 去重新生成**。
   - 但因为依旧会超时，因此永远都会失败；甚至在后台，多次点击会向 DeepSeek 发送多条并行的请求，白白消耗大量 Token 且极易触发并发限制。
4. **缓存被“毒化”**（上一步中我已为你初步修复）：
   - 如果某个半成品被存入了 `bookCache`，代码会永久返回这个没有全文的残次品。

## 🛠️ 彻底解决的架构重构方案 (Proposed Changes)

为了彻底解决长耗时请求导致的断连问题，我建议实施一套**“状态锁 + 客户端静默轮询”**机制。

### 1. 云端超时配置放宽 [已实施]
- 为 `bookService` 和 `aiGenerator` 分别创建 `config.json`，将服务端执行超时上限设定为最高 **60秒**，防止后端因超时强制杀死进程。

### 2. 后端生成状态锁（避免重复调用耗费Token）[需实施]
- 修改 `cloudfunctions/bookService/index.js`。
- 在决定调用 AI 之前，先将数据库里该书的 `aiFullContent` 标记为 `["内容生成中..."]` 写入缓存作为**“互斥锁”**。
- 如果有其他请求进来，发现 `aiFullContent` 还是 `["内容生成中..."]`，**不再重复发起 AI 请求**，而是直接返回一个明确的标志：`{ success: true, isGenerating: true }`。

### 3. 前端静默轮询机制 (Polling) [需实施]
- 修改 `miniprogram/pages/detail/detail.js`。
- 当 `fetchBookDetail` 收到 `isGenerating: true` 或遭遇**客户端网络超时**时，前端不再粗暴地把 `aiLoading` 设置为 false。
- 前端将维持 `aiLoading: true` 的视觉状态，并在后台启动一个定时器，**每隔 3-5 秒去静默查询一次**（因为后端可能还在努力生成），直到拿到了完整的 `aiFullContent` 才关闭动画并允许用户阅读。
- 如果彻底失败，再提示用户并允许手动重试。
